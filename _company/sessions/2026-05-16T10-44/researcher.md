## 의료기기 규제 준수 상세 체크리스트 (BLE 통신 및 위험 알림 기능 중심)

본 체크리스트는 Developer가 설계한 BLE 통신 보안 아키텍처와 위험 구간 알림 기능이 SaMD(Software as a Medical Device) 프레임워크 및 관련 데이터 보호 규정(HIPAA, GDPR 등)을 준수하는지 검토하기 위해 개발 방향을 제시합니다.

### 섹션 1: 의료기기 분류 및 범위 정의 (SaMD 프레임워크 기반)

**목표:** 애플리케이션의 기능이 어떤 의료기기로 분류되는지를 명확히 하고, 이에 따른 규제 요구사항의 범위를 설정한다.

| 항목 | 검토 내용 | 준수 여부 (Y/N/NA) | 근거/증빙 자료 | 개발 영향도 |
| :--- | :--- | :--- | :--- | :--- |
| **1.1 기기 분류 확인** | 본 애플리케이션의 기능(데이터 수집, 위험 알림)이 SaMD 정의에 따라 Class I, II, III 중 어디에 해당하는지 명확히 정의했는가? | | | 높음 |
| **1.2 인허가 요구사항 매핑** | 분류에 따라 요구되는 의료기기 허가/인증 절차(예: FDA 510(k) 또는 EU MDR)의 초기 단계 요구사항을 파악했는가? | | | 높음 |
| **1.3 위험 등급 정의** | '위험 구간 알림' 기능이 환자의 생명이나 건강에 미치는 영향을 기반으로 임상적 위험 등급(Critical, Serious, Minor)을 명확히 분류하고 시스템에 반영했는가? | | | 높음 |

### 섹션 2: BLE 통신 보안 아키텍처 검증 (P0 요구사항 중심)

**목표:** P0로 정의된 데이터 암호화, 인증/인가, 불변 감사 로그의 기술적 구현이 규제 기준을 충족하는지 확인한다.

| 항목 | 검토 내용 | 준수 여부 (Y/N/NA) | 근거/증빙 자료 | 개발 영향도 |
| :--- | :--- | :--- | :--- | :--- |
| **2.1 데이터 암호화** | BLE 통신 중(In-transit) 및 저장 시(At-rest)의 모든 민감 데이터(혈당, 센서 값, 사용자 정보)가 AES-256 이상의 표준을 사용하여 암호화되었는가? | | | 필수 |
| **2.2 인증/인가 메커니즘** | BLE Link Layer 수준에서 상호 인증(Mutual Authentication) 및 권한 기반 접근 제어(Access Control) 메커니즘이 구현되었는가? (BLE Secure Connections 활용 여부 포함) | | | 필수 |
| **2.3 불변 감사 로그** | 모든 위험 이벤트 발생, 데이터 접근 시도, 설정 변경 기록이 위변조 방지 기술(해시 체인 또는 블록체인/Secure Storage)을 통해 안전하게 기록되고 보존되는가? | | | 필수 |
| **2.4 보안 취약점 분석** | 설계된 통신 모듈에 대해 정적/동적 분석(SAST/DAST)을 수행하여 잠재적인 BLE 관련 공격 벡터 및 취약점을 식별했는가? | | | 높음 |

### 섹션 3: 데이터 프라이버시 및 규제 준수 (HIPAA/GDPR 중심)

**목표:** 수집된 데이터의 민감도에 따라 적절한 접근 통제 및 보존 정책이 적용되었는지 검증한다.

| 항목 | 검토 내용 | 준수 여부 (Y/N/NA) | 근거/증빙 자료 | 개발 영향도 |
| :--- | :--- | :--- | :--- | :--- |
| **3.1 데이터 최소화 원칙** | 위험 알림 및 기능 수행에 **필수적인 최소한의 데이터**만 수집하고 전송하는가? 불필요한 개인 식별 정보(PII)는 분리하여 처리하는가? | | | 높음 |
| **3.2 동의 관리 (Consent)** | 사용자로부터 데이터 수집 및 사용에 대한 명시적 동의(Explicit Consent)를 획득하고, 이 동의 상태를 시스템이 실시간으로 추적하며 반영하는가? | | | 필수 |
| **3.3 접근 통제** | 환자의 민감한 의료 데이터에 대한 접근 권한은 최소 권한 원칙(Principle of Least Privilege)에 따라 엄격하게 관리되고 로그로 기록되는가? | | | 높음 |
| **3.4 데이터 보존 및 삭제** | 규제 요구사항에 따른 데이터의 안전한 보존 기간 및 사용자 요청 시 완전하고 복구 불가능한 방식으로 데이터를 삭제하는 절차가 정의되어 있는가? | | | 중간 |

### 섹션 4: 임상적 타당성 및 알림 시스템 검증 (P0 위험 알림 중심)

**목표:** 위험 알림 기능이 기술적으로 정확할 뿐만 아니라, 임상적 맥락에서 의미 있는 결과를 제공하는지 검증한다.

| 항목 | 검토 내용 | 준수 여부 (Y/N/NA) | 근거/증빙 자료 | 개발 영향도 |
| :--- | :--- | :--- | :--- | :--- |
| **4.1 임계값의 임상적 타당성** | 위험 알림에 사용되는 임계값(Thresholds)이 실제 의료 환경에서의 생리학적 반응 및 의학적 지식에 기반하여 설정되었는가? (단순 센서 값이 아닌 맥락 고려) | | | 필수 |
| **4.2 오탐 방지 메커니즘** | 센서 오류, 네트워크 불안정 등 외부 요인으로 인한 '오탐(False Alarm)'을 방지하기 위한 교차 검증(Redundancy Check) 및 안전장치(Fail-safe)가 구현되었는가? | | | 필수 |
| **4.3 알림의 명확성** | 시스템이 생성하는 위험 알림 메시지가 환자 또는 의료진에게 상황의 심각도와 조치 사항을 명확하고 오해 없이 전달하는가? (UI/UX 검증 필요) | | | 높음 |
