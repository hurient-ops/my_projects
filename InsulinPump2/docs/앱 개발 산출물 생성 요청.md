## **2\. 제품 비전 및 목적**

본 제품은 대한민국 내 1형 당뇨병 환자의 인슐린 펌프 사용률을 높이고, 목표 범위 내 시간(Time In Range, TIR)을 기존 $46.9%$에서 자동 모드 적용 시 ![][image1] 이상으로 개선하는 데 기여하고자 한다.8 사용자가 자신의 질병 관리 데이터를 주도적으로 활용하게 함으로써 자기효능감을 증진시키는 것이 핵심 비전이다.1

## **4\. 기능적 요구사항**

기능 설계는 'In\_pump ver 0.80' 프로토콜에서 정의된 투약 로직을 충실히 반영한다.7

### **4.1 핵심 주입 제어 (Injection Control)**

애플리케이션은 아침, 점심, 저녁 식사 주입과 추가 주입(Bolus)을 지원해야 한다. 특히 BT\_INJ\_REQ (0x17) 메시지를 통해 주입을 요청하며, 주입 시작(BT\_INJ\_START\_IND, 0x3A)과 완료(BT\_INJ\_STOP\_IND, 0x3B) 상태를 실시간으로 UI에 피드백해야 한다.7

### **4.2 상황별 특수 모드 (Specialized Modes)**

사용자의 생활 패턴에 따른 기초량(Basal) 변화를 반영하기 위해 다음과 같은 모드를 제공한다.

* **운동 모드 (Exercise Mode):** 기초 주입 중 운동 시작을 알리고 감량 단위(20단위)를 적용하여 저혈당을 예방한다.7  
* **회식 모드 (Reception Mode):** 장시간 지속되는 식사 상황에 맞춰 주입 시간 및 양을 조정한다.7

### **4.3 기기 상태 및 이력 관리**

* **실시간 모니터링:** 5단계 배터리 잔량(0x71) 및 인슐린 잔량을 상시 표시한다.7  
* **데이터 동기화:** 펌프에 저장된 최대 15일간의 이력 데이터를 BT\_LOG\_REQ (0x1D)를 통해 가져와 차트로 시각화한다.7

의료기기 소프트웨어로서의 신뢰성을 위해 성능 및 보안 기준을 설정한다.

## **1\. 서론**

본 기술 명세서는 스마트폰과 인슐린 펌프 간의 안정적인 실시간 데이터 교환을 구현하기 위한 아키텍처를 정의한다. 20-byte 고정 길이 메시지 구조를 가지는 'In\_pump' 프로토콜을 모바일 환경의 리액티브 데이터 스트림으로 변환하는 것이 기술적 핵심이다.7

## **2\. 시스템 아키텍처**

애플리케이션은 계층화된 아키텍처(Layered Architecture)를 채택하여 하드웨어 의존성을 분리한다.

* **Hardware Interface Layer:** 블루투스 스택 관리 및 원시 바이트 데이터 송수신 (flutter\_blue\_plus 기반).23  
* **Protocol Transformation Layer:** 20-byte 패킷의 직렬화/역직렬화 및 Little-endian 변환 처리.7  
* **Domain Logic Layer:** 기초량 계산 로직, 투약 제한(Limit) 검증, 상태 머신 관리.  
* **Presentation Layer:** 상태 관리 라이브러리(Bloc/Riverpod)를 통한 반응형 UI 구현.

## **3\. 통신 프로토콜 상세 분석**

프로토콜 명세서 ver 0.80에 따른 주요 패킷 구조를 기술한다.7

### **3.1 공통 메시지 구조**

모든 메시지는 20바이트 고정이며, Little-endian 형식을 사용한다.7

| Byte Offset | 필드명 | 설명 | 값/예시 |
| :---- | :---- | :---- | :---- |
| 0 | Start Code | 메시지의 시작 알림 | 0xEF |
| 1 | Type Code | 메시지의 목적 (ID) | 예: 0x17 (주입 요청) |
| 2 | Data Length | 실제 데이터 부분의 길이 | 0 \~ 17 |
| 3 \~ 19 | Data Payload | 메시지별 상세 데이터 | 가변 데이터 |

### **3.2 핵심 상태 및 응답 코드 (RES\_CODE)**

요청에 대한 기기의 처리 가능 여부를 확인하기 위해 1-byte 응답 코드를 분석한다.7

* **RES\_OK (0x00):** 요청 처리 가능.  
* **RES\_INVALID\_STATUS (0x01):** 현재 기기 상태(예: 주입 중)에서 요청 처리 불가.  
* **RES\_INVALID\_PARAM (0x02):** 매개변수 값 오류.  
* **RES\_CAN\_NOT\_HANDLE\_MSG (0x04):** 기타 처리 불가.

### **3.3 시간대별 기초 설정 (BT\_TIME\_BASE\_SET\_REQ, 0x0F)**

버전 0.80의 주요 변경사항으로, 24시간 기초량을 4시간 단위로 나누어 총 6회에 걸쳐 데이터를 전송한다. 각 전송 시 12바이트의 데이터를 포함하며, 이는 환자의 활동 주기에 따른 정밀 투여를 가능케 한다.7

## **5\. 데이터 보안 아키텍처**

의료기기 사이버보안 허가 심사 기준(2025-2026)에 따라 다중 방어 전략을 수립한다.28

* **암호화 (Encryption):** 정지 상태 데이터(AES-256)와 전송 상태 데이터(TLS 1.3) 모두 강력한 암호 알고리즘을 적용한다.22

* **가용성 (Availability):** 기기 연결 해제 상황 발생 시 즉각적인 재연결 로직과 로컬 알람 기능을 통해 주입 중단 위험을 방지한다.17

    Clogged \--\> Log\[에러 이력 저장 및 의료진 공유\]

**문서 4: AI 마일스톤 기반 프롬프트 설계 (Prompt Design)**

## **목차**

1. 서론  
2. 마일스톤 1: BLE 기저 통신 및 보안 핸드쉐이크  
3. 마일스톤 2: 바이너리 프로토콜 직렬화 및 상태 머신 구축  
4. 마일스톤 3: 주입 제어 로직 및 임상 안전 가드레일  
5. 마일스톤 4: 대량 이력 데이터 동기화 및 가공  
6. 마일스톤 5: 에러 진단 엔진 및 규제 대응 고도화

## **1\. 서론**

본 설계는 AI 코딩 에이전트(예: Claude Code, GPT-4o)가 실제 작동 가능한 코드를 생성할 수 있도록, 명확한 제약 조건과 아키텍처 가이드라인을 5단계 마일스톤으로 제시한다. 코드 샘플을 직접 제공하지 않고 프로토콜의 논리적 구조와 데이터 흐름에 집중한다.14

## **2\. 마일스톤 1: BLE 기저 통신 및 보안 핸드쉐이크**

**목표:** 하드웨어 검색, 연결 유지, 그리고 펌프와의 초기 인증 절차를 완결한다.

**AI 프롬프트 가이드:**

"인슐린 펌프 'Heal Us' 기기를 위한 BLE 연결 계층을 구축하십시오.

1. 서비스 이름 'Heal Us'를 필터링하는 스캔 로직을 구현하십시오.  
2. 연결 직후 20바이트 패킷 \[0xEF, 0x02, 0x00,...\]을 송신하여 BT\_CONNECTABLE\_CTRL\_REQ를 수행하십시오.  
3. 비밀번호 '000000'을 BT\_PRS\_APP\_PASSWD\_IND (0x3C) 메시지에 실어 보내는 인증 로직을 설계하십시오.  
4. 모든 블루투스 작업은 5초 타임아웃을 적용하고, 연결 해제 시 기하급수적 백오프(Exponential Backoff) 전략으로 재연결을 시도하십시오.  
5. 보안을 위해 LE Secure Connections(Mode 1, Level 4\) 설정을 우선적으로 요청하도록 구성하십시오."

## **3\. 마일스톤 2: 바이너리 프로토콜 직렬화 및 상태 머신 구축**

**목표:** 20바이트 고정 길이 메시지를 앱 내 객체로 변환하고 전역 상태를 관리한다.

**AI 프롬프트 가이드:**

"In\_pump ver 0.80 명세서에 따른 패킷 직렬화 엔진을 개발하십시오.

1. 모든 메시지는 20바이트 Uint8List로 고정되며, 0번 바이트는 항상 0xEF여야 합니다.  
2. 정수 데이터는 Little-endian 방식을 적용하여 바이트 배열로 변환하는 유틸리티를 작성하십시오.  
3. BT\_STATE\_IND (0x05) 및 BT\_BATT\_DATA\_IND (0x71) 수신 시 앱의 전역 상태(상태 관리 프레임워크 사용)를 업데이트하십시오.  
4. 앱이 포그라운드에 있는 동안 30초마다 BT\_STATE\_REQ (0x04)를 송신하는 하트비트(Heartbeat) 메커니즘을 추가하십시오.  
5. 수신된 데이터의 길이가 2번 바이트(Data Length)와 일치하지 않을 경우 해당 패킷을 폐기하는 검증 로직을 포함하십시오."

## **4\. 마일스톤 3: 주입 제어 로직 및 임상 안전 가드레일**

**목표:** 인슐린 투여 프로세스를 구현하고 사용자 실수에 의한 과다 복용을 방지하는 안전 장치를 마련한다.

**AI 프롬프트 가이드:**

"가장 중요한 기능인 인슐린 주입(Injection) 워크플로우를 구현하십시오.

1. BT\_INJ\_REQ (0x17) 메시지 생성 시 inj\_sel (식사=0, 추가=1)과 2바이트 inj\_val을 정확히 배치하십시오.  
2. 주입 명령 전, 현재 펌프의 잔량과 설정 한도를 BT\_INJ\_INFO\_REQ (0x15)를 통해 사전 확인하는 '이중 확인(Double-Check)' 프로세스를 설계하십시오.  
3. 주입 중 긴급 상황을 위해 BT\_INJ\_STOP\_REQ (0x37)를 최우선 순위로 송신하는 기능을 구현하십시오.  
4. UI 레벨에서 최종 '주입 시작' 버튼은 사용자의 의도적 동작(예: 슬라이드 투 컨펌 또는 롱 프레스)을 요구해야 합니다."

## **5\. 마일스톤 4: 대량 이력 데이터 동기화 및 가공**

**목표:** 펌프의 내부 메모리에 기록된 과거 데이터를 안정적으로 대량 전송받아 분석한다.

**AI 프롬프트 가이드:**

"15일치 주입 이력 데이터를 동기화하는 모듈을 구축하십시오.

1. BT\_LOG\_REQ (0x1D)를 송신한 후, 기기로부터 오는 대량 패킷 스트림을 처리하십시오.  
2. 스트림의 시작(0x0B)과 종료(0x0C)를 감지하여 데이터 무결성을 확인하십시오.  
3. 각 14바이트 로그 엔트리를 파싱하여 로컬 암호화 데이터베이스에 저장하십시오 (필드: 날짜, 기초량 총합, 식사량 총합, 추가 주입량 등).  
4. 동기화 진행률을 사용자에게 퍼센트 단위로 표시하고, 중단 시 마지막 수신 지점부터 재개하는 로직을 고려하십시오."

## **6\. 마일스톤 5: 에러 진단 엔진 및 규제 대응 고도화**

**목표:** 11가지 에러 상태를 처리하고 규제 기관 제출을 위한 로그 및 보안 요구사항을 충족한다.

**AI 프롬프트 가이드:**

"의료기기 품질 기준에 부합하도록 애플리케이션을 강화하십시오.

1. BT\_ERR\_IND (0x19) 수신 시 11가지 에러 코드(바늘 막힘, 모터 이상, 배터리 부족 등)를 사용자 친화적인 메시지와 조치 가이드로 변환하는 매핑 엔진을 만드십시오.  
2. 모든 투약 요청과 기기 응답을 타임스탬프와 함께 기록하는 '변조 불가능한 감사 로그(Audit Trail)'를 구현하십시오.  
3. 앱 실행 시 루팅/탈옥 여부를 탐지하고, 비정상 환경에서 주입 기능을 차단하는 보안 가드 로직을 추가하십시오.  
4. 2026년 MFDS 가이드라인에 따라 앱 배경화면 전환 시 민감한 의료 데이터가 노출되지 않도록 스냅샷 보호 기능을 적용하십시오."
